# ADR-002: 외부 API 클라이언트 위치 결정

## 날짜
2026-05-14

## 상태
`결정됨`

---

## 상황

현재 Kakao API와 통신하는 클라이언트 클래스 두 개가 도메인 패키지 루트에 위치한다.

```
gift/auth/
  KakaoLoginClient.java    # Kakao OAuth 토큰 발급, 유저 정보 조회
gift/order/
  KakaoMessageClient.java  # Kakao 메시지 발송
```

이 클래스들은 도메인 로직이 아닌 **외부 시스템과의 HTTP 통신**을 담당한다.
도메인 패키지 안에 두면 "인증 로직"과 "Kakao API 호출 방법"이 같은 계층으로 취급되어,
외부 API가 변경될 때 도메인 패키지를 직접 수정해야 한다.

---

## 선택지

| 선택지 | 설명 | 장점 | 단점 |
|--------|------|------|------|
| A. 현행 유지 | 도메인 패키지 루트에 클라이언트 보관 | 변경 없음 | 외부 통신과 도메인 로직의 경계가 없음, 책임 불명확 |
| B. 최상위 `gift.infra` 패키지 | 모든 외부 클라이언트를 `gift.infra.kakao.*`로 통합 | 인프라 클래스를 한 곳에서 관리, 외부 의존 파악 쉬움 | 도메인과 인프라 관계가 패키지 구조에서 드러나지 않음 |
| C. 도메인 하위 `infra/` 서브패키지 | `gift.auth.infra.KakaoLoginClient`, `gift.order.infra.KakaoMessageClient` | 도메인 응집도 유지 + 인프라 계층 명시 | 패키지 depth 증가 |

---

## 결정

**B를 선택한다. 최상위 `gift.infra` 패키지에 외부 클라이언트를 모은다.**

외부 API 클라이언트는 특정 도메인의 소유가 아니라 **애플리케이션이 외부 세계와 통신하는 방법**이다.
Kakao 로그인 클라이언트가 꼭 `auth` 도메인 소유라고 볼 수 없고,
추후 Kakao 외 다른 외부 API가 추가될 경우에도 `infra` 아래에 일관되게 배치할 수 있다.

```
gift/infra/
  kakao/
    KakaoLoginClient.java    # OAuth 토큰 발급, 유저 정보 조회
    KakaoMessageClient.java  # 메시지 발송
```

서비스(컨트롤러)는 `gift.infra.kakao.*`를 주입받아 사용한다.

---

## 트레이드오프

감수하는 것: 어떤 도메인이 어떤 외부 클라이언트를 쓰는지 패키지 구조만으로는 알 수 없음

얻는 것: 외부 통신 코드가 한 곳에 모여 외부 API 변경 영향 범위 파악이 쉬움,
도메인 패키지가 순수하게 도메인 로직만 담게 됨

---

## 결과

- `gift.auth.KakaoLoginClient` → `gift.infra.kakao.KakaoLoginClient`
- `gift.order.KakaoMessageClient` → `gift.infra.kakao.KakaoMessageClient`
- 두 클래스를 import하는 컨트롤러(`KakaoAuthController`, `OrderController`)의 import 경로 수정 필요
- 이후 외부 API 클라이언트(결제, 알림 등) 추가 시 `gift.infra.*` 하위에 일관되게 배치
